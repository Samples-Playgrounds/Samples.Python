# Comprehensive Report: .NET Agent Frameworks and Libraries (2026)

## Executive Summary

The .NET agent framework landscape underwent a major consolidation in April 2026 with Microsoft's release of **Agent Framework 1.0**, which unified the previously separate Semantic Kernel and AutoGen projects into a single production-ready SDK. This represents the current state-of-the-art for enterprise .NET agent development, alongside community alternatives like LangChain for .NET.

---

## 1. Microsoft Agent Framework 1.0 (Production-Ready, April 2026)

### Overview
Microsoft Agent Framework 1.0 reached general availability on April 3, 2026, marking the production-ready milestone for Microsoft's unified approach to AI agent development [1]. This framework represents the convergence of over 75,000 combined GitHub stars from Semantic Kernel and AutoGen into a single, enterprise-grade SDK [5].

### Key Features & Capabilities

**Core Architecture (5-Layer Design)** [5]:
1. **Connectors** - Provider-specific adapters for multiple LLM services
2. **Kernel** - DI container and configuration surface (inherited from Semantic Kernel)
3. **Agents** - First-class agent primitive with instructions, tools, memory, and state
4. **Orchestration** - Graph workflow engine for multi-agent patterns
5. **Interop Layer** - MCP and A2A protocol adapters for external integration

**Stable 1.0 Features** [1]:
- Single-agent abstraction and service connectors (both .NET and Python)
- Middleware hooks for extensibility
- Agent memory and context providers
- Graph-based workflows
- Multi-agent orchestration patterns: sequential, concurrent, handoff, group chat, and Magentic-One
- Long-term support commitment with backward compatibility guarantees

**Preview Features** [1]:
- DevUI - Browser-based local debugger for real-time agent execution inspection
- Foundry hosted agent integration
- Front-end adapters for AG-UI, CopilotKit, and ChatKit
- Reusable "skills" system
- GitHub Copilot SDK and Claude Code SDK integration
- Agent Harness for shell, file system, and messaging-loop access

### LLM Provider Support

Agent Framework 1.0 ships with first-party connectors for [1][5]:
- **Microsoft Foundry**
- **Azure OpenAI** (with Managed Identity and API key auth)
- **OpenAI**
- **Anthropic Claude**
- **Amazon Bedrock** (IAM auth)
- **Google Gemini** (API key/OAuth)
- **Ollama** (local models)

All providers support streaming responses, and swapping providers requires only a single-line configuration change [5].

### Multi-Agent Orchestration Patterns

The framework provides four canonical orchestration patterns out of the box [5]:

| Pattern | Use Case |
|---------|----------|
| **Round-robin** | Each agent has a clear turn in a deterministic pipeline |
| **Supervisor** | One coordinator routes messages to specialist agents and aggregates outputs |
| **Hierarchical** | Tree topology with managers owning sub-teams; useful for complex research tasks |
| **Dynamic hand-off** | Agents decide at runtime which peer to transfer to |

### Interoperability Standards

**Model Context Protocol (MCP)** [5]:
- Full native support for dynamic tool discovery
- Agents can attach any MCP server (Google Ads MCP, GitHub MCP, custom servers)
- Tools appear automatically without code changes as server catalogs evolve

**Agent-to-Agent Protocol (A2A)** [1][5]:
- A2A 1.0 support enables cross-framework agent coordination
- Allows Agent Framework agents to coordinate with agents from other frameworks (Hermes Agent, LangChain, custom implementations)
- Structured, protocol-driven messaging with consistent wire format

### Code Example (.NET)

```csharp
// Basic agent setup
using Microsoft.Agents.AI;
using Microsoft.Agents.AI.OpenAI;

var agent = new ChatAgent(
    name: "Concierge",
    instructions: "You are a concise agency concierge. Answer in one paragraph.",
    model: new OpenAIChatClient("gpt-4.1", Environment.GetEnvironmentVariable("OPENAI_API_KEY")!)
);

await foreach (var token in agent.RunStreamingAsync("Explain MCP in 3 sentences."))
{
    Console.Write(token.Text);
}
```

```csharp
// MCP integration example
var agent = new ChatAgent("TradingAgent", instructions: "...")
    .WithMcpServer("https://mcp.example.com/trading")
    .WithMcpServer(new StdioMcpServer("pipx", "run", "--spec", "...", "google-ads-mcp"));
```

### Platform Support & Deployment

- **Languages**: .NET 9+ (C# 13), Python 3.10+ [5]
- **Package**: `Microsoft.Agents.AI` (.NET) / `microsoft-agents-ai` (Python) [5]
- **License**: MIT [5]
- **Deployment**: Azure App Service reference architecture with managed identity support [5]
- **Observability**: Native OpenTelemetry emission for production monitoring [5]

### Best Suited For
- Enterprise applications requiring long-term support and stability
- Multi-agent systems with complex orchestration requirements
- Teams already invested in Microsoft Azure ecosystem
- Applications requiring cross-framework agent interoperability via A2A
- Production deployments needing backward compatibility guarantees

### Community & Maturity
- **Status**: Production-ready (1.0 GA as of April 2026)
- **Support**: Long-term support commitment from Microsoft
- **Migration Path**: Documented upgrade paths from both Semantic Kernel and AutoGen [1]

## 2. Semantic Kernel (Foundation Layer, Active Maintenance)

### Overview
Semantic Kernel is Microsoft's lightweight, open-source SDK that now serves as the foundation layer for Microsoft Agent Framework [2]. It remains actively maintained with recent releases continuing to appear in its GitHub repository [1].

### Key Features & Capabilities

**Core Capabilities** [2]:
- **Model-Agnostic Design**: Easily connects to latest AI models; swap models without rewriting code
- **Plugin System**: Integrates existing APIs as callable functions for AI models
- **Middleware Architecture**: Hooks and filters for extensibility
- **Multi-Language Support**: C#, Python, and Java with version 1.0+ support
- **Enterprise-Ready**: Used by Microsoft and Fortune 500 companies [2]

**Architecture Components** [2]:
- Kernel abstraction for AI orchestration
- Plugin model for function integration
- Prompt template language
- Service connectors for chat, embeddings, and memory
- Telemetry support for observability
- Security-enhancing hooks and filters

### Integration Capabilities
Semantic Kernel uses OpenAPI specifications (similar to Microsoft 365 Copilot), allowing extensions to be shared with pro-code and low-code developers [2].

### Code Example

```csharp
// Semantic Kernel remains the foundation for plugin development
// Plugins can be shared across Agent Framework implementations
```

### Best Suited For
- Teams building single-agent applications
- Developers who need a stable, proven foundation without complex multi-agent orchestration
- Projects requiring plugin-based extensibility
- Applications that need to integrate existing APIs with AI models
- Cross-platform development (C#, Python, Java)

### Current Status
- **Maintenance**: Active, with ongoing releases [1]
- **Relationship to Agent Framework**: Serves as the foundation layer; not deprecated [1]
- **Migration**: Can be lazy - migrate as features require [5]

## 3. AutoGen (Maintenance Mode)

### Overview
AutoGen is Microsoft's open-source framework for multi-agent AI applications, now in maintenance mode with bug fixes and critical security patches only [1]. Its orchestration concepts have been rebuilt into Microsoft Agent Framework 1.0 [5].

### Key Features & Capabilities

**Core Components** [3]:
- **UserProxyAgent**: Human gateway for injections, clarifications, or approvals at trust boundaries
- **AssistantAgent**: Handles fully autonomous reasoning with chosen LLM
- **ConversableAgent**: Base class for all agent types
- **GroupChat Manager**: Coordinates multi-agent conversations without hard-coded pipelines [3]

**Architecture** [3]:
- **Conversation-First Design**: Treats everything as a conversation; agents send structured chat messages
- **Asynchronous Execution**: Agents run asynchronously across containers or nodes
- **Lightweight Scheduler**: Determines speaking order automatically
- **Sandboxed Code Execution**: Python runner with restricted file system and network access [3]

**Orchestration Mechanics** [3]:
- Speaker-selection logic based on pending tasks, message history, or custom heuristics
- Termination rules: max rounds, explicit "DONE" tokens, or satisfaction checks
- Audit logs for all messages flowing through orchestrator
- Support for sharding multiple GroupChat instances behind load balancer

### Tool Integration
- Declarative tool integration (agents reference "calculator" or "sql_db")
- AutoGen handles argument parsing and result injection
- Execution policy stored in `code_execution_config` [3]

### Model Support
Model-agnostic with support for OpenAI, Azure, Anthropic, and local models via JSON configuration [3].

### Best Suited For
- Legacy applications already built on AutoGen (maintenance only)
- Teams planning migration to Agent Framework during 2026 [5]

### Current Status
- **Maintenance Mode**: Bug fixes and critical security patches only [1]
- **Migration Path**: Microsoft recommends moving to Agent Framework 1.0 [1][5]
- **Timeline**: Plan AutoGen migrations during 2026 [5]

## 4. LangChain for .NET (Community-Driven Alternative)

### Overview
LangChain for .NET is a community-driven, open-source C# implementation of LangChain, maintained by the tryAGI organization [4]. It represents an alternative to Microsoft's frameworks with a focus on maximum choice and third-party library integration.

### Key Features & Capabilities

**Core Features** [4]:
- **Composability**: Building applications with LLMs through composable components
- **Chains**: Sequential and complex workflow orchestration
- **Vector Databases**: Built-in support for document embeddings and similarity search
- **Document Loaders**: PDF, text, and other document format support
- **Text Splitters**: Configurable chunking (default: ChunkSize=4000, ChunkOverlap=200)
- **Memory Systems**: Context management for conversational agents
- **Dependency Injection**: Native .NET DI support

**Provider Support** [4]:
- OpenAI
- Azure OpenAI
- Anthropic
- DeepSeek
- Google (Gemini)
- HuggingFace
- Ollama (local models)

**Database Support** [4]:
- SQLite vector database (built-in)
- In-memory databases
- Extensible to other vector stores

### Platform Support
- **.NET Versions**: .NET 8.0, .NET 9.0, .NET Standard 2.0, .NET Framework 4.6.2 [4]
- **Package**: `LangChain` (NuGet)
- **Current Version**: 0.17.0 (stable, released January 31, 2025) [4]
- **License**: MIT [4]

### Code Example

```csharp
// RAG example with vector database
using LangChain;
using LangChain.Databases.Sqlite;
using LangChain.DocumentLoaders.Pdf;

// Initialize models
var provider = new OpenAiProvider(Environment.GetEnvironmentVariable("OPENAI_API_KEY"));
var llm = new OpenAiLatestFastChatModel(provider);
var embeddingModel = new TextEmbeddingV3SmallModel(provider);

// Create vector database from PDF
using var vectorDatabase = new SqLiteVectorDatabase(dataSource: "vectors.db");
var vectorCollection = await vectorDatabase.AddDocumentsFromAsync<PdfPigPdfLoader>(
    embeddingModel,
    dimensions: 1536,
    dataSource: DataSource.FromUrl("https://example.com/document.pdf"),
    collectionName: "documents",
    textSplitter: null);

// Query with chains
const string question = "What is the main topic?";
var similarDocuments = await vectorCollection.GetSimilarDocuments(
    embeddingModel, question, amount: 5);

var answer = await llm.GenerateAsync($"""
    Use the following context to answer the question.
    {similarDocuments.AsString()}
    Question: {question}
    Answer:
    """);

// Alternative: Use chain syntax
var chain =
    Set("What is the main topic?")
    | RetrieveSimilarDocuments(vectorCollection, embeddingModel, amount: 5)
    | CombineDocuments(outputKey: "context")
    | Template(promptTemplate)
    | LLM(llm.UseConsoleForDebug());
    
var chainAnswer = await chain.RunAsync("text");
```

## 6. Use Case Recommendations

### Choose **Microsoft Agent Framework 1.0** when:
- Building production multi-agent systems requiring enterprise support
- Need cross-framework interoperability via A2A protocol
- Require long-term support and backward compatibility guarantees
- Working within Microsoft Azure ecosystem
- Building complex orchestration with supervisor, hierarchical, or dynamic hand-off patterns
- Need native MCP support for dynamic tool discovery

### Choose **Semantic Kernel** when:
- Building single-agent applications with plugin extensibility
- Need cross-platform support (C#, Python, Java)
- Require a stable, proven foundation without complex orchestration
- Want to integrate existing APIs as AI-callable functions
- Building on a mature, enterprise-tested framework
- Planning eventual migration to Agent Framework

### Choose **LangChain for .NET** when:
- Building RAG applications with heavy document processing
- Need built-in vector database support
- Prefer community-driven, vendor-neutral solutions
- Want Python LangChain-like patterns in .NET
- Require extensive text splitting and embedding capabilities
- Comfortable with community support model

### Avoid **AutoGen** for new projects:
- Now in maintenance mode; migrate to Agent Framework 1.0 instead
- Only use for maintaining existing AutoGen applications
- Plan migration during 2026 per Microsoft guidance

## 7. Emerging Trends & Future Outlook

### Protocol Standardization
The emergence of MCP (Model Context Protocol) and A2A (Agent-to-Agent) as native features in Agent Framework 1.0 signals industry movement toward standardized agent interoperability [1][5]. This allows .NET agents to coordinate with agents built in other frameworks and languages.

### Consolidation Around Agent Framework
Microsoft's unification of Semantic Kernel and AutoGen into Agent Framework 1.0 represents a strategic consolidation, providing a single recommended path for new .NET agent development [1][5]. This reduces fragmentation in the Microsoft ecosystem.

### Community Alternatives Remain Viable
LangChain for .NET demonstrates that community-driven alternatives continue to thrive, particularly for specialized use cases like RAG and document processing where they offer more opinionated, batteries-included solutions [4].

### DevOps & Observability Maturation
The inclusion of DevUI for local debugging and native OpenTelemetry support in Agent Framework 1.0 shows the ecosystem maturing toward production-grade observability and developer experience [5].

## 8. Migration Guidance

### From AutoGen to Agent Framework 1.0
- **Timeline**: Plan migrations during 2026 [5]
- **Approach**: AutoGen's orchestration concepts are rebuilt as graph workflows in Agent Framework
- **Key Changes**: GroupChat patterns map to orchestration patterns (round-robin, supervisor, etc.)
- **Resources**: Microsoft provides documented migration guides [1]

### From Semantic Kernel to Agent Framework 1.0
- **Timeline**: Can be lazy - migrate as features require [5]
- **Approach**: Semantic Kernel becomes the foundation layer; existing plugins remain compatible
- **Key Changes**: Add orchestration layer for multi-agent scenarios
- **Compatibility**: High - Semantic Kernel concepts carry forward

### From LangChain .NET to Agent Framework 1.0
- **Consideration**: Only if enterprise support and multi-agent orchestration are required
- **Trade-offs**: Lose built-in RAG features; gain enterprise backing and A2A/MCP support
- **Approach**: Rebuild chains as graph workflows; migrate vector DB to Azure AI Search or similar

## Conclusion

The .NET agent framework landscape in 2026 is dominated by **Microsoft Agent Framework 1.0**, which represents the production-ready convergence of Semantic Kernel and AutoGen. For enterprise applications, this is the recommended choice with its long-term support, multi-agent orchestration, and native MCP/A2A interoperability.

**Semantic Kernel** remains relevant as both a foundation layer and a standalone option for single-agent scenarios, while **AutoGen** is in maintenance mode pending migration. **LangChain for .NET** offers a compelling community-driven alternative, particularly for RAG-focused applications and teams preferring vendor-neutral solutions.

The key decision factors are:
1. **Enterprise vs. Community Support**: Microsoft frameworks offer LTS; LangChain offers flexibility
2. **Single vs. Multi-Agent**: Semantic Kernel for single; Agent Framework for multi
3. **Specialized Needs**: LangChain excels at RAG and document processing
4. **Interoperability**: Agent Framework 1.0 leads with MCP and A2A support
