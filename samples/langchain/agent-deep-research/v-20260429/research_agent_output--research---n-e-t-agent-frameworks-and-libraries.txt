# Comprehensive Report: .NET Agent Frameworks and Libraries (2026)

## Executive Summary

The .NET agent framework landscape underwent a major consolidation in April 2026 with Microsoft's release of **Agent Framework 1.0**, which unified the previously separate Semantic Kernel and AutoGen projects into a single production-ready SDK. This represents the current state-of-the-art for enterprise .NET agent development, alongside community alternatives like LangChain for .NET.

---

## 1. Microsoft Agent Framework 1.0 (Production-Ready, April 2026)

### Overview
Microsoft Agent Framework 1.0 reached general availability on April 3, 2026, marking the production-ready milestone for Microsoft's unified approach to AI agent development [1]. This framework represents the convergence of over 75,000 combined GitHub stars from Semantic Kernel and AutoGen into a single, enterprise-grade SDK [5].

### Key Features & Capabilities

**Core Architecture (5-Layer Design)** [5]:
1. **Connectors** - Provider-specific adapters for multiple LLM services
2. **Kernel** - DI container and configuration surface (inherited from Semantic Kernel)
3. **Agents** - First-class agent primitive with instructions, tools, memory, and state
4. **Orchestration** - Graph workflow engine for multi-agent patterns
5. **Interop Layer** - MCP and A2A protocol adapters for external integration

**Stable 1.0 Features** [1]:
- Single-agent abstraction and service connectors (both .NET and Python)
- Middleware hooks for extensibility
- Agent memory and context providers
- Graph-based workflows
- Multi-agent orchestration patterns: sequential, concurrent, handoff, group chat, and Magentic-One
- Long-term support commitment with backward compatibility guarantees

**Preview Features** [1]:
- DevUI - Browser-based local debugger for real-time agent execution inspection
- Foundry hosted agent integration
- Front-end adapters for AG-UI, CopilotKit, and ChatKit
- Reusable "skills" system
- GitHub Copilot SDK and Claude Code SDK integration
- Agent Harness for shell, file system, and messaging-loop access

### LLM Provider Support

Agent Framework 1.0 ships with first-party connectors for [1][5]:
- **Microsoft Foundry**
- **Azure OpenAI** (with Managed Identity and API key auth)
- **OpenAI**
- **Anthropic Claude**
- **Amazon Bedrock** (IAM auth)
- **Google Gemini** (API key/OAuth)
- **Ollama** (local models)

All providers support streaming responses, and swapping providers requires only a single-line configuration change [5].

### Multi-Agent Orchestration Patterns

The framework provides four canonical orchestration patterns out of the box [5]:

| Pattern | Use Case |
|---------|----------|
| **Round-robin** | Each agent has a clear turn in a deterministic pipeline |
| **Supervisor** | One coordinator routes messages to specialist agents and aggregates outputs |
| **Hierarchical** | Tree topology with managers owning sub-teams; useful for complex research tasks |
| **Dynamic hand-off** | Agents decide at runtime which peer to transfer to |

### Interoperability Standards

**Model Context Protocol (MCP)** [5]:
- Full native support for dynamic tool discovery
- Agents can attach any MCP server (Google Ads MCP, GitHub MCP, custom servers)
- Tools appear automatically without code changes as server catalogs evolve

**Agent-to-Agent Protocol (A2A)** [1][5]:
- A2A 1.0 support enables cross-framework agent coordination
- Allows Agent Framework agents to coordinate with agents from other frameworks (Hermes Agent, LangChain, custom implementations)
- Structured, protocol-driven messaging with consistent wire format

### Code Example (.NET)

```csharp
// Basic agent setup
using Microsoft.Agents.AI;
using Microsoft.Agents.AI.OpenAI;

var agent = new ChatAgent(
    name: "Concierge",
    instructions: "You are a concise agency concierge. Answer in one paragraph.",
    model: new OpenAIChatClient("gpt-4.1", Environment.GetEnvironmentVariable("OPENAI_API_KEY")!)
);

await foreach (var token in agent.RunStreamingAsync("Explain MCP in 3 sentences."))
{
    Console.Write(token.Text);
}
```

```csharp
// MCP integration example
var agent = new ChatAgent("TradingAgent", instructions: "...")
    .WithMcpServer("https://mcp.example.com/trading")
    .WithMcpServer(new StdioMcpServer("pipx", "run", "--spec", "...", "google-ads-mcp"));
```

### Platform Support & Deployment

- **Languages**: .NET 9+ (C# 13), Python 3.10+ [5]
- **Package**: `Microsoft.Agents.AI` (.NET) / `microsoft-agents-ai` (Python) [5]
- **License**: MIT [5]
- **Deployment**: Azure App Service reference architecture with managed identity support [5]
- **Observability**: Native OpenTelemetry emission for production monitoring [5]

### Best Suited For
- Enterprise applications requiring long-term support and stability
- Multi-agent systems with complex orchestration requirements
- Teams already invested in Microsoft Azure ecosystem
- Applications requiring cross-framework agent interoperability via A2A
- Production deployments needing backward compatibility guarantees

### Community & Maturity
- **Status**: Production-ready (1.0 GA as of April 2026)
- **Support**: Long-term support commitment from Microsoft
- **Migration Path**: Documented upgrade paths from both Semantic Kernel and AutoGen [1]

## 2. Semantic Kernel (Foundation Layer, Active Maintenance)

### Overview
Semantic Kernel is Microsoft's lightweight, open-source SDK that now serves as the foundation layer for Microsoft Agent Framework [2]. It remains actively maintained with recent releases continuing to appear in its GitHub repository [1].

### Key Features & Capabilities

**Core Capabilities** [2]:
- **Model-Agnostic Design**: Easily connects to latest AI models; swap models without rewriting code
- **Plugin System**: Integrates existing APIs as callable functions for AI models
- **Middleware Architecture**: Hooks and filters for extensibility
- **Multi-Language Support**: C#, Python, and Java with version 1.0+ support
- **Enterprise-Ready**: Used by Microsoft and Fortune 500 companies [2]

**Architecture Components** [2]:
- Kernel abstraction for AI orchestration
- Plugin model for function integration
- Prompt template language
- Service connectors for chat, embeddings, and memory
- Telemetry support for observability
- Security-enhancing hooks and filters

### Integration Capabilities
Semantic Kernel uses OpenAPI specifications (similar to Microsoft 365 Copilot), allowing extensions to be shared with pro-code and low-code developers [2].

### Code Example

```csharp
// Semantic Kernel remains the foundation for plugin development
// Plugins can be shared across Agent Framework implementations
```

### Best Suited For
- Teams building single-agent applications
- Developers who need a stable, proven foundation without complex multi-agent orchestration
- Projects requiring plugin-based extensibility
- Applications that need to integrate existing APIs with AI models
- Cross-platform development (C#, Python, Java)

### Current Status
- **Maintenance**: Active, with ongoing releases [1]
- **Relationship to Agent Framework**: Serves as the foundation layer; not deprecated [1]
- **Migration**: Can be lazy - migrate as features require [5]

## 3. AutoGen (Maintenance Mode)

### Overview
AutoGen is Microsoft's open-source framework for multi-agent AI applications, now in maintenance mode with bug fixes and critical security patches only [1]. Its orchestration concepts have been rebuilt into Microsoft Agent Framework 1.0 [5].

### Key Features & Capabilities

**Core Components** [3]:
- **UserProxyAgent**: Human gateway for injections, clarifications, or approvals at trust boundaries
- **AssistantAgent**: Handles fully autonomous reasoning with chosen LLM
- **ConversableAgent**: Base class for all agent types
- **GroupChat Manager**: Coordinates multi-agent conversations without hard-coded pipelines [3]

**Architecture** [3]:
- **Conversation-First Design**: Treats everything as a conversation; agents send structured chat messages
- **Asynchronous Execution**: Agents run asynchronously across containers or nodes
- **Lightweight Scheduler**: Determines speaking order automatically
- **Sandboxed Code Execution**: Python runner with restricted file system and network access [3]

**Orchestration Mechanics** [3]:
- Speaker-selection logic based on pending tasks, message history, or custom heuristics
- Termination rules: max rounds, explicit "DONE" tokens, or satisfaction checks
- Audit logs for all messages flowing through orchestrator
- Support for sharding multiple GroupChat instances behind load balancer

### Tool Integration
- Declarative tool integration (agents reference "calculator" or "sql_db")
- AutoGen handles argument parsing and result injection
- Execution policy stored in `code_execution_config` [3]

### Model Support
Model-agnostic with support for OpenAI, Azure, Anthropic, and local models via JSON configuration [3].

### Best Suited For
- Legacy applications already built on AutoGen (maintenance only)
- Teams planning migration to Agent Framework during 2026 [5]

### Current Status
- **Maintenance Mode**: Bug fixes and critical security patches only [1]
- **Migration Path**: Microsoft recommends moving to Agent Framework 1.0 [1][5]
- **Timeline**: Plan AutoGen migrations during 2026 [5]

## 4. LangChain for .NET (Community-Driven Alternative)

### Overview
LangChain for .NET is a community-driven, open-source C# implementation of LangChain, maintained by the tryAGI organization [4]. It represents an alternative to Microsoft's frameworks with a focus on maximum choice and third-party library integration.

### Key Features & Capabilities

**Core Features** [4]:
- **Composability**: Building applications with LLMs through composable components
- **Chains**: Sequential and complex workflow orchestration
- **Vector Databases**: Built-in support for document embeddings and similarity search
- **Document Loaders**: PDF, text, and other document format support
- **Text Splitters**: Configurable chunking (default: ChunkSize=4000, ChunkOverlap=200)
- **Memory Systems**: Context management for conversational agents
- **Dependency Injection**: Native .NET DI support

**Provider Support** [4]:
- OpenAI
- Azure OpenAI
- Anthropic
- DeepSeek
- Google (Gemini)
- HuggingFace
- Ollama (local models)

**Database Support** [4]:
- SQLite vector database (built-in)
- In-memory databases
- Extensible to other vector stores

### Platform Support
- **.NET Versions**: .NET 8.0, .NET 9.0, .NET Standard 2.0, .NET Framework 4.6.2 [4]
- **Package**: `LangChain` (NuGet)
- **Current Version**: 0.17.0 (stable, released January 31, 2025) [4]
- **License**: MIT [4]

### Code Example

```csharp
// RAG example with vector database
using LangChain;
using LangChain.Databases.Sqlite;
using LangChain.DocumentLoaders.Pdf;

// Initialize models
var provider = new OpenAiProvider(Environment.GetEnvironmentVariable("OPENAI_API_KEY"));
var llm = new OpenAiLatestFastChatModel(provider);
var embeddingModel = new TextEmbeddingV3SmallModel(provider);

// Create vector database from PDF
using var vectorDatabase = new SqLiteVectorDatabase(dataSource: "vectors.db");
var vectorCollection = await vectorDatabase.AddDocumentsFromAsync<PdfPigPdfLoader>(
    embeddingModel,
    dimensions: 1536,
    dataSource: DataSource.FromUrl("https://example.com/document.pdf"),
    collectionName: "documents",
    textSplitter: null);

// Query with chains
const string question = "What is the main topic?";
var similarDocuments = await vectorCollection.GetSimilarDocuments(
    embeddingModel, question, amount: 5);

var answer = await llm.GenerateAsync($"""
    Use the following context to answer the question.
    {similarDocuments.AsString()}
    Question: {question}
    Answer:
    """);

// Alternative: Use chain syntax
var chain =
    Set("What is the main topic?")
    | RetrieveSimilarDocuments(vectorCollection, embeddingModel, amount: 5)
    | CombineDocuments(outputKey: "context")
    | Template(promptTemplate)
    | LLM(llm.UseConsoleForDebug());
    
var chainAnswer = await chain.RunAsync("text");
```

- Developers preferring community-driven, vendor-neutral solutions
- Applications needing extensive document processing and vector search
- Teams comfortable with community support vs. enterprise backing

- Active development with frequent updates (dev builds in March-April 2026) [4]

## 6. Use Case Recommendations

### Choose **Microsoft Agent Framework 1.0** when:
- Building production multi-agent systems requiring enterprise support
- Need cross-framework interoperability via A2A protocol
- Require long-term support and backward compatibility guarantees
- Working within Microsoft Azure ecosystem
- Building complex orchestration with supervisor, hierarchical, or dynamic hand-off patterns
- Need native MCP support for dynamic tool discovery

### Choose **Semantic Kernel** when:
- Building single-agent applications with plugin extensibility
- Need cross-platform support (C#, Python, Java)
- Require a stable, proven foundation without complex orchestration
- Want to integrate existing APIs as AI-callable functions
- Building on a mature, enterprise-tested framework
- Planning eventual migration to Agent Framework

### Choose **LangChain for .NET** when:
- Building RAG applications with heavy document processing
- Need built-in vector database support
- Prefer community-driven, vendor-neutral solutions
- Want Python LangChain-like patterns in .NET
- Require extensive text splitting and embedding capabilities
- Comfortable with community support model

### Avoid **AutoGen** for new projects:
- Now in maintenance mode; migrate to Agent Framework 1.0 instead
- Only use for maintaining existing AutoGen applications
- Plan migration during 2026 per Microsoft guidance

## 7. Emerging Trends & Future Outlook

### Protocol Standardization
The emergence of MCP (Model Context Protocol) and A2A (Agent-to-Agent) as native features in Agent Framework 1.0 signals industry movement toward standardized agent interoperability [1][5]. This allows .NET agents to coordinate with agents built in other frameworks and languages.

### Consolidation Around Agent Framework
Microsoft's unification of Semantic Kernel and AutoGen into Agent Framework 1.0 represents a strategic consolidation, providing a single recommended path for new .NET agent development [1][5]. This reduces fragmentation in the Microsoft ecosystem.

### Community Alternatives Remain Viable
LangChain for .NET demonstrates that community-driven alternatives continue to thrive, particularly for specialized use cases like RAG and document processing where they offer more opinionated, batteries-included solutions [4].

### DevOps & Observability Maturation
The inclusion of DevUI for local debugging and native OpenTelemetry support in Agent Framework 1.0 shows the ecosystem maturing toward production-grade observability and developer experience [5].

## 8. Migration Guidance

### From AutoGen to Agent Framework 1.0
- **Timeline**: Plan migrations during 2026 [5]
- **Approach**: AutoGen's orchestration concepts are rebuilt as graph workflows in Agent Framework
- **Key Changes**: GroupChat patterns map to orchestration patterns (round-robin, supervisor, etc.)
- **Resources**: Microsoft provides documented migration guides [1]

### From Semantic Kernel to Agent Framework 1.0
- **Timeline**: Can be lazy - migrate as features require [5]
- **Approach**: Semantic Kernel becomes the foundation layer; existing plugins remain compatible
- **Key Changes**: Add orchestration layer for multi-agent scenarios
- **Compatibility**: High - Semantic Kernel concepts carry forward

### From LangChain .NET to Agent Framework 1.0
- **Consideration**: Only if enterprise support and multi-agent orchestration are required
- **Trade-offs**: Lose built-in RAG features; gain enterprise backing and A2A/MCP support
- **Approach**: Rebuild chains as graph workflows; migrate vector DB to Azure AI Search or similar

## Conclusion

The .NET agent framework landscape in 2026 is dominated by **Microsoft Agent Framework 1.0**, which represents the production-ready convergence of Semantic Kernel and AutoGen. For enterprise applications, this is the recommended choice with its long-term support, multi-agent orchestration, and native MCP/A2A interoperability.

**Semantic Kernel** remains relevant as both a foundation layer and a standalone option for single-agent scenarios, while **AutoGen** is in maintenance mode pending migration. **LangChain for .NET** offers a compelling community-driven alternative, particularly for RAG-focused applications and teams preferring vendor-neutral solutions.

The key decision factors are:
1. **Enterprise vs. Community Support**: Microsoft frameworks offer LTS; LangChain offers flexibility
2. **Single vs. Multi-Agent**: Semantic Kernel for single; Agent Framework for multi
3. **Specialized Needs**: LangChain excels at RAG and document processing
4. **Interoperability**: Agent Framework 1.0 leads with MCP and A2A support

# Comprehensive Report: .NET Agent Frameworks and Libraries

## Executive Summary

The .NET agent framework landscape is experiencing a major consolidation in 2025-2026. Microsoft has merged its two primary frameworks—**Semantic Kernel** and **AutoGen**—into a unified **Microsoft Agent Framework** (currently in Release Candidate status, with GA expected Q1 2026). This consolidation eliminates the confusion that plagued developers choosing between overlapping Microsoft offerings and creates a single, enterprise-grade solution for building AI agents in .NET [1].

Unlike the Python ecosystem where multiple competing frameworks (LangGraph, CrewAI, LangChain) serve different niches, the .NET space is dominated by Microsoft's offerings, which makes strategic sense given Microsoft's ownership of the .NET platform itself.

**Official Name:** Microsoft Agent Framework  
**Status:** Release Candidate (announced at .NET Conf 2025), GA expected Q1 2026

#### Core Purpose and Description

Microsoft Agent Framework represents the convergence of Semantic Kernel's enterprise orchestration capabilities with AutoGen's multi-agent conversation patterns. It provides a unified SDK for building AI agents with first-class support for multi-agent workflows, human-in-the-loop patterns, and enterprise integration [1][4].

The framework treats **agents as first-class concepts** rather than requiring developers to wire up services and build kernels manually. This architectural shift simplifies agent creation while maintaining the flexibility needed for complex enterprise scenarios.

**Agent Creation & Management:**
- Simple agent instantiation with name, instructions, and capabilities
- Type-safe function tool definitions using `AIFunction.Create()`
- Native streaming support for real-time responses
- Built-in human-in-the-loop approval workflows [4]

**Multi-Agent Orchestration Patterns:**
- **Sequential Workflows:** Agents execute in defined order (writer → reviewer → finalizer)
- **Concurrent Workflows:** Multiple agents work in parallel with aggregated results
- **Handoff Patterns:** Dynamic agent-to-agent transfer with routing logic
- **Group Chat:** Multi-agent conversations with configurable speaker selection [4]

**Integration Capabilities:**
- **Model Context Protocol (MCP):** Native support for MCP-compatible tool servers
- **Agent-to-Agent (A2A) Protocol:** Interoperability standard for cross-framework communication

**Enterprise Features:**
- Production SLAs and formal support contracts
- SOC 2, HIPAA compliance guarantees
- Role-based access control (RBAC)
- Audit logging and security guarantees
- Deep Azure integration (Azure OpenAI, Azure SQL, Azure services) [2]

### 2. Semantic Kernel (Legacy - Being Merged)

**Official Name:** Semantic Kernel  
**Status:** Mature, transitioning to Microsoft Agent Framework

#### Core Purpose and Description

Semantic Kernel is Microsoft's SDK for integrating Large Language Models into .NET applications. It focuses on **orchestration, plugins, and enterprise integration** rather than pure multi-agent conversations. The framework enables developers to encapsulate existing APIs as plugins that AI can invoke through function calling [3].

While Semantic Kernel continues to be maintained, Microsoft is actively migrating developers to the unified Agent Framework. Existing Semantic Kernel applications remain supported, but new development should target Agent Framework [4].

#### Key Features and Capabilities

**Plugin System:**
- Class-based plugins with `[KernelFunction]` attributes
- Semantic descriptions for AI-friendly function discovery
- Dependency injection support for complex services
- 100+ data connectors for enterprise systems [3]

**Function Calling:**
- Native LLM function calling support
- Automatic parameter marshaling and type conversion
- Return type schema generation for accurate AI responses
- Few-shot examples and detailed function descriptions [3]

**Orchestration:**

public class LightsPlugin
{
   private readonly List<LightModel> lights = new()

   {
      return lights;
   }

      light.Hex = LightModel.Hex;

### 3. AutoGen / AutoGen.NET (Legacy - Being Merged)

**Official Name:** AutoGen (Microsoft Research project)  
**Status:** Mature research framework, merging into Microsoft Agent Framework

#### Core Purpose and Description

AutoGen is Microsoft Research's framework for building **multi-agent conversational systems**. Unlike Semantic Kernel's focus on orchestration and plugins, AutoGen treats agent-to-agent communication as natural language conversations. This conversation-first design eliminates brittle API pipelines and makes complex workflows easier to prototype [5].

AutoGen pioneered the multi-agent patterns that are now being incorporated into the unified Microsoft Agent Framework, including group chats, speaker selection strategies, and human-in-the-loop workflows [2][5].

#### Key Features and Capabilities

**Agent Types:**
- **UserProxyAgent:** Human gateway for clarifications and approvals
- **AssistantAgent:** Autonomous reasoning with LLM integration
- **ConversableAgent:** Base class for custom agent implementations [5]

**Conversation Orchestration:**
- **GroupChat Manager:** Coordinates multi-agent conversations without hard-coded pipelines
- **Speaker Selection Logic:** Custom heuristics for determining turn order
- **Termination Rules:** Max rounds, explicit tokens, satisfaction checks to prevent infinite loops [5]

**Code Execution:**
- Sandboxed Python runner for secure code execution
- Restricted file system and network access
- Declarative tool integration with argument parsing
- Resource limits and package whitelists [5]

#### Architectural Patterns

**Conversation-First Design:**
- All coordination happens through structured chat messages
- Natural language handoffs replace custom RPC calls
- Transparent, replayable traces for debugging
- Asynchronous agent execution across containers [5]

**State Management:**
- Each agent maintains independent context windows
- Tool permissions isolated per agent
- Configuration via JSON or environment variables
- Docker/Kubernetes deployment support [5]

## Architectural Patterns and Best Practices

### 1. Plugin/Tool Design Patterns

**Semantic Kernel Approach:**
```csharp
public class WeatherPlugin
{
    [KernelFunction("get_weather")]
    [Description("Gets the current weather for a location")]
    public string GetWeather([Description("The city name")] string city)
    {
        return $"The weather in {city} is sunny, 72°F";
    }
}
```

**Agent Framework Approach:**
```csharp
var weatherTool = AIFunction.Create(
    [Description("Gets the current weather for a location")]
    (string city) => $"The weather in {city} is sunny, 72°F",
    name: "get_weather"
);
```

**Best Practices:**
- Use descriptive, concise function names (avoid abbreviations)
- Minimize function parameters (use primitive types when possible)
- Provide detailed descriptions only when necessary to reduce token consumption
- Use snake_case for function names (LLMs trained on Python conventions)
- Implement proper error handling and validation [3]

### 2. Multi-Agent Orchestration Patterns

**Sequential Workflow:**
- Use when agents must execute in specific order
- Example: Research → Analysis → Writing → Review
- Best for linear, dependent tasks [4]

**Concurrent Workflow:**
- Use when agents can work independently
- Example: Multiple research agents gathering different data sources
- Best for parallel, independent tasks [4]

**Handoff Pattern:**
- Use when agents need dynamic routing
- Example: Customer service (sales → support → billing)
- Requires router agent for decision-making [4]

**Group Chat:**
- Use for collaborative problem-solving
- Example: Brainstorming, debate, consensus-building
- Requires speaker selection strategy [5]

### 3. State Management Strategies

**Centralized State:**
- Single source of truth for agent context
- Prevents desynchronization issues
- Requires conflict resolution logic [5]

**Checkpointing:**
- Persist state after critical transitions
- Enable rollback on failures
- Support long-running workflows [5]

**Partitioning:**
- Isolate state by task or customer ID
- Reduce lock contention
- Improve throughput under load [5]

Agents, converting plugins to tools using `AIFunction.Create()`, updating orchestration to workflow patterns, and testing incrementally [4].\n\n### Community and Maturity\n\nSemantic Kernel is a mature framework with over 2 years in production use. The framework features excellent documentation through comprehensive Microsoft Learn resources with tutorials, best practices, and migration guides. The active GitHub community provides extensive sample repositories and tutorials, and there is a clear migration path to the Agent Framework [3][4].\n\n## AutoGen\n\nAutoGen is Microsoft Research\'s framework for building multi-agent conversational systems. Unlike Semantic Kernel\'s focus on orchestration and plugins, AutoGen treats agent-to-agent communication as natural language conversations. This conversation-first design eliminates brittle API pipelines and makes complex workflows easier to prototype [5].\n\nAutoGen pioneered the multi-agent patterns that are now being incorporated into the unified Microsoft Agent Framework, including group chats, speaker selection strategies, and human-in-the-loop workflows [2][5].\n\n### Key Features\n\n**Agent Types:**\nThe framework provides UserProxyAgent as a human gateway for clarifications and approvals, AssistantAgent for autonomous reasoning with LLM integration, and ConversableAgent as a base class for custom agent implementations [5].\n\n**Conversation Orchestration:**\nGroupChat Manager coordinates multi-agent conversations without hard-coded pipelines. Speaker Selection Logic provides custom heuristics for determining turn order. Termination Rules include max rounds, explicit tokens, and satisfaction checks to prevent infinite loops [5].\n\n**Code Execution:**\nThe framework includes a sandboxed Python runner for secure code execution with restricted file system and network access, declarative tool integration with argument parsing, and resource limits and package whitelists [5].\n\n### Architectural Patterns\n\nAutoGen\'s conversation-first design means all coordination happens through structured chat messages. Natural language handoffs replace custom RPC calls, providing transparent, replayable traces for debugging and enabling asynchronous agent execution across containers [5].\n\nState management in AutoGen involves each agent maintaining independent context windows, tool permissions isolated per agent, configuration via JSON or environment variables, and Docker/Kubernetes deployment support [5].\n\n### Code Example\n\nMulti-agent setup in AutoGen demonstrates the conversation-first approach:\n\n```python\nfrom autogen import UserProxyAgent, AssistantAgent, GroupChat, GroupChatManager\n\n# Create agents\nuser_proxy = UserProxyAgent(\n    name="user_proxy",\n    human_input_mode="TERMINATE",\n    code_execution_config={"work_dir": "coding"}\n)\n\nassistant = AssistantAgent(\n    name="assistant",\n    llm_config={"model": "gpt-4", "api_key": "..."}\n)\n\n# Group chat orchestration\ngroupchat = GroupChat(\n    agents=[user_proxy, assistant],\n    messages=[],\n    max_round=10\n)\n\nmanager = GroupChatManager(groupchat=groupchat, llm_config={"model": "gpt-4"})\n```\n\n### Supported LLM Providers\n\nAutoGen supports OpenAI (GPT-4, GPT-3.5), Azure OpenAI Service, Anthropic Claude, local models via compatible APIs, and custom LLM endpoints [5].\n\n### Use Cases\n\nAutoGen is best suited for research and experimentation with multi-agent systems, conversational AI requiring natural agent interactions, prototyping complex workflows before production deployment, and teams familiar with AutoGen who are migrating to the Agent Framework [2][5].\n\n### Challenges and Limitations\n\nProduction deployment with AutoGen faces several challenges. Non-deterministic conversations mean the same prompts can trigger different dialogue paths, complicating debugging. State synchronization issues arise because isolated agent memory buffers can lead to coordination failures. Resource contention occurs when GPU cycles and API quotas cause conversation timeouts. Distributed debugging becomes difficult with hundreds of interleaved LLM calls that are hard to trace without proper tooling [5].\n\nThe Agent Framework addresses these challenges through centralized state management with checkpointing, workflow-based orchestration with explicit control flow, resource scheduling and adaptive throttling, and distributed tracing with correlation IDs [5].\n\n### Community and Maturity\n\nAutoGen is a mature research framework with over 30,000 GitHub stars (combined Python and .NET). It has a strong research community with active development from Microsoft Research. The framework features good research-oriented documentation with examples and tutorials. While smaller than the Semantic Kernel community, it is highly engaged, and there is a clear migration path to the Agent Framework [2][5].\n\n## Framework Comparison\n\nWhen comparing the three frameworks across key dimensions, several patterns emerge:\n\n**Status and Focus:**\nMicrosoft Agent Framework is in Release Candidate status (GA expected Q1 2026) and serves as a unified agent platform. Minimize function parameters and use primitive types when possible. Provide detailed descriptions only when necessary to reduce token consumption. Use snake_case for function names since LLMs are trained on Python conventions. Implement proper error handling and validation [3].\n\nSemantic Kernel plugins follow this pattern:\n\n```csharp\npublic class WeatherPlugin\n{\n    [KernelFunction("get_weather")]\n    [Description("Gets the current weather for a location")]\n    public string GetWeather([Description("The city name")] string city)\n    {\n        return $"The weather in {city} is sunny, 72°F";\n    }\n}\n```\n\nAgent Framework tools use a more concise approach:\n\n```csharp\nvar weatherTool = AIFunction.Create(\n    [Description("Gets the current weather for a location")]\n    (string city) => $"The weather in {city} is sunny, 72°F",\n    name: "get_weather"\n);\n```\n\n### Multi-Agent Orchestration\n\nSequential workflows should be used when agents must execute in a specific order, such as Research → Analysis → Writing → Review. This pattern is best for linear, dependent tasks [4].\n\nConcurrent workflows are appropriate when agents can work independently, such as multiple research agents gathering different data sources. This pattern is best for parallel, independent tasks [4].\n\nHandoff patterns should be used when agents need dynamic routing, such as customer service scenarios involving sales → support → billing transitions. This pattern requires a router agent for decision-making [4].\n\nGroup chat is suitable for collaborative problem-solving scenarios like brainstorming, debate, and consensus-building. This pattern requires a speaker selection strategy [5].\n\n### State Management\n\nCentralized state provides a single source of truth for agent context, prevents desynchronization issues, but requires conflict resolution logic [5].\n\nCheckpointing persists state after critical transitions, enables rollback on failures, and supports long-running workflows [5].\n\nPartitioning isolates state by task or customer ID, reduces lock contention, and improves throughput under load [5].\n\n## Ecosystem Gaps\n\nUnlike the Python ecosystem, the .NET space lacks mature ports of popular frameworks. ## Key Findings:

**Major Consolidation in Progress:**
Microsoft is merging Semantic Kernel and AutoGen into a unified **Microsoft Agent Framework** (RC status, GA expected Q1 2026). This eliminates the confusion of choosing between overlapping frameworks.

**Three Main Frameworks:**

1. **Microsoft Agent Framework** (Recommended for new projects)
   - Unified platform combining best of Semantic Kernel + AutoGen
   - Native multi-agent workflows (Sequential, Concurrent, Handoff, Group Chat)
   - Enterprise features: SLAs, RBAC, SOC 2/HIPAA compliance
   - Multi-language support: C#, Python, Java
   - Native MCP (Model Context Protocol) support

2. **Semantic Kernel** (Mature, transitioning)
   - Focus on orchestration, plugins, and enterprise integration
   - 100+ data connectors
   - Strong RAG capabilities
   - Clear migration path to Agent Framework

3. **AutoGen** (Mature research framework, transitioning)
   - Conversation-first multi-agent design
   - GroupChat orchestration
   - Best for research and prototyping
   - Migrating to Agent Framework for production

**Ecosystem Characteristics:**
- Microsoft monoculture (unlike diverse Python ecosystem)
- No mature .NET ports of LangGraph, CrewAI, or LangChain
- Strong Azure integration bias
- Enterprise-grade support and compliance

**Recommendation:** Start new projects with Microsoft Agent Framework despite RC status—it's stable and represents Microsoft's strategic direction.
